## 主要问题

### P1 炸金花结算积分非零和

5 人满座测试中，结算前均为 1000 分，结算后为：-20、-20、-40、+140、-80，合计 -20。  
疑似原因在 `server/index.js` 的 `settleGameWithPot`：赢家按 `totalPot - playerBet` 加分，输家按各自 `playerBet` 扣分，但实测 `pot` 与 `playerBets` 汇总不一致时会吞分。

相关位置：`server/index.js:1459`、`server/index.js:1465`、`server/index.js:1469`、`server/index.js:1471`

### P1 炸金花 4 人无法开局

当前配置允许炸金花 `minPlayers: 2, maxPlayers: 5`，但候场房启动条件使用“满员”判断，4 个玩家全部准备后仍不会开始，最终准备超时/启动超时。  
如果产品期望“2-5 人均可开局”，需要把准备开始条件从 `isRoomFull(room)` 调整为达到最小人数并全员准备，或在 UI 明确必须 5 人。

相关位置：`server/game-engine.js:1864`、`server/index.js:633`、`server/index.js:647`、`server/index.js:737`

### P1 象棋难以自然结算，且结算字段存在高风险

自动对弈 90 秒、570 步后仍停在 `playing`，积分不变化。更重要的是代码层面象棋引擎写入的赢家是颜色值 `red/black`，而通用结算逻辑按玩家 ID 判断赢家；一旦自然结束，存在双方都不被识别为赢家的风险。

相关位置：`server/game-engines/chess.js:342`、`server/game-engines/chess.js:363`、`server/index.js:1290`、`server/index.js:1311`

### P2 掼蛋长局无法在测试窗口内完成结算

补测中已自动触发 `round_finished -> next_round`，但 90 秒/603 步后仍处于 `play`，积分不变化。可能是正常长局，也可能是机器人策略或局数推进缺少可测的快速收敛条件。建议增加测试模式的回合上限或强制结算路径，否则积分系统很难稳定回归。

### P2 手机观战页显示问题

- 观战页显示“当前观战人数: 0”，即使专家已通过 `spectator:join` 加入。
- 五子棋/象棋观战页没有棋盘状态展示，仅显示玩家列表和游戏 key。

相关位置：`client/src/views/Spectator.vue:156`、`client/src/views/Spectator.vue:172`、`server/index.js:417`

## 积分系统结论

已自然结算的剪刀石头布、快问快答、五子棋、斗地主、红中麻将均为零和结算，符合预期。  
炸金花存在明确非零和问题，需要优先修复。象棋与掼蛋未完成自然结算，积分正确性仍有残余风险。

## 建议

1. 优先修复炸金花 pot/playerBets 结算一致性，并补一条单元测试校验“所有参与者积分 delta 合计必须为 0”。
2. 明确私有候场房的开局规则：按 `minPlayers` 开始，或 UI 文案和按钮状态明确“必须满员”。
3. 象棋结算统一使用玩家 ID，不要把 `red/black` 直接交给通用积分结算。
4. 为掼蛋、象棋增加可测试的短局/强制结算模式，方便积分回归。
5. 补齐观战页的五子棋/象棋展示，并让服务端返回或广播真实观战人数。
